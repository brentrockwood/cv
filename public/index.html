2004-2012 Quest Software/Dell Halifax, Nova Scotia
Senior Software Developer

  • Quest One Identity Manager - Data Governance Edition

    I worked on this product team for two years. Our team in Halifax ranged from ten to twenty developers and testers, collaborating with similar sized teams in Germany, Russia, China, and across the US. Our team was primarily focused on filesystem and SharePoint security, as well as categorization and classification of untyped filesystem data (documents). Our components alone comprised over a million lines of code and one hundred code projects. Every developer on the team was required to understand and own all of the code we produced.

    All work items were tracked using Microsoft Team Foundation Server (TFS), and we operated in two week sprints, culminating in biweekly demos which were viewed by stakeholders around the world.

    All check-ins to TFS source control were subject to mandatory code review and continuously integrated using MSBuild and Team Build.

    Unit tests used MSTest. Integration testing used SpecFlow and custom PowerShell scripts. Performance testing also used PowerShell and custom tools to collect and publish performance counters. All results of testing were continuously published to all stakeholders.

    The core Windows service component was built using C#. Communication with other components was primarily via Windows Communications Foundation and REST interfaces. Highlights of the service included a custom threading module for handling job execution with more control than the Microsoft-provided thread pool. The data layer of the service supported both MS SQL Server and Oracle.

    User interfaces were built using Windows Forms, and for the web, ASP.Net, HTML5 and jQuery. All user interfaces were fully internationalized for translation into multiple end-user languages. PowerShell commandlets were also provided and documented for end-user scripting.

    Security scanning agent processes were installed on target machines via a custom remote executor. The agent itself was comprised of a C++ core, with a COM plugin architecture for data scanners. The scanners included out of the box could scan NTFS filesystems and SharePoint for security data. In addition, the agent could receive change events from both sources. In the case of SharePoint, this used a combination of the SharePoint event API, and custom code which was designed by reverse engineering the SharePoint database itself. In the case of the filesystem, this used a custom Windows kernel driver, written in C++, and validated by Microsoft. The agent also used an SQLite database for local storage.

    Agents were also able to tokenize and push filesystem and SharePoint data to classification services which would read the unstructured data and make a determination if it contained Personally Identifiable Information (PII), Protected Health Information (PHI), or Payment Card Industry (PCI) data.

    All Windows installers were built using the Windows Installer XML (WiX) framework.

  • Quest Collaboration Services

    I worked on this product team for one year. I took over this product as technical lead after a previous team had difficulty maintaining it at the required minimum service level. The product was designed to synchronize contact and calendar data between Microsoft Exchange servers without any security changes or trusts required between the participating organizations, as in a large contracting situation, for example. Synchronization was entirely via encrypted email.

    The product had been built in Russia by a team that no longer existed. Due to a problem during a code migration before I received the product, all comments and documentation were lost from the code, which weighed in at about 150,000 significant lines. In addition, the product, written entirely in C++, used undocumented and unsupported MAPI interfaces to communicate with Exchange.

    Working with one other developer and two testers, we were able to release versions extending support for Exchange 2010, along with numerous bug-fix releases. In addition, we provided 24-7 support to all customers around the world.

    Practices and processes were broadly similar to those listed above, with the notable exception of testing, as it was not deemed economically feasible to build a full automated test suite for the legacy code.

  • Quest Access Manager

    I worked on this product team for one year. Access Manager was the predecessor to the Data Governance product listed above. Many of the components were reused by the later product. Access Manager did not contain the SharePoint or data classification components, and was a standalone product with a user interface built into Microsoft Management console (MMC).

  • Quest Group Policy Manager

    I worked on this product team for two years. Group Policy Manager was the predecessor to a product now known as Quest GPOAdmin. Starting from scratch, and with significant competition in the marketplace, we built it into the world's leading third-party Group Policy management tool. It was primarily a Group Policy version control tool, offering policy comparison, workflow, and deployment features. The team was comprised of four developers and one tester.

    A client-server design, most code was written in C#, with some C++ for policy setting injection. We heavily leveraged the API's provided by Microsoft with their Group Policy Management Console. For functionality that could not be provided by the public API's, we reverse engineered the GPO structures in Active Directory.

    The user interface was presented as an MMC snap-in, with HTML used for reporting purposes. All UI was translated into French, German, and Japanese. The client communicated via .Net Remoting to a Windows service. The service, in turn, used a data layer that supported filesystem storage, SQL Server, or Active Directory itself.

    This project predated common use of Scrum, Agile, and related methodologies. As a result, it was managed much more informally. However, we did use continuous integration and constant code review to help maintain schedules and quality.
